# Devpost submission copy

**Project name:** Sockeye - Agentic SOC Triage on the Splunk MCP Server<br>
**Track:** Security<br>
**Opt-in category:** Best Use of Splunk MCP Server<br>
**Repository:** https://github.com/dmetagame/sockeye

**Elevator pitch:** Sockeye is an autonomous SOC analyst that investigates
security incidents in Splunk, using the official Splunk MCP Server as a
least-privilege, auditable interface to operational evidence.

## Inspiration

Tier-1 triage is some of the highest-volume work in a SOC: establish whether an
alert is real, determine whether access succeeded, scope follow-on activity, and
hand a responder a defensible containment plan. Many AI security demos summarize
an alert that another system already created. We wanted an agent that performs
the investigation itself: form a hypothesis, write SPL, inspect results, pivot,
and preserve the evidence behind its conclusion.

## What it does

Sockeye connects Claude to a live Splunk Enterprise instance through the
official Splunk MCP Server (Splunkbase 7931). On each run it:

1. sweeps the selected index and time window by sourcetype and volume,
2. measures authentication failures by source and account,
3. pivots from suspicious attempts to successful access and later behavior,
4. correlates escalation, lateral movement, collection, and outbound transfer,
5. emits a markdown report with verdict, severity, timeline, IOCs, evidence, and
   ranked response actions, and
6. appends a query ledger generated by the runner from the exact MCP tool calls.

The included 1,976-event scenario contains normal activity plus a distributed
password spray, a focused brute-force attack against `svc-backup`, successful
access, `sudo su -`, movement to a file server, sensitive-data staging, and
570,904,170 bytes of outbound transfer. The prompt describes the investigation
method, not the planted answer.

## How we built it

- Splunk Enterprise 10.4 runs in a pinned Docker environment.
- Splunk MCP Server 1.2.0 exposes the read-only `splunk_run_query` tool used by
  the agent over MCP.
- Claude Agent SDK for Python drives the investigation loop with an explicit
  model, strict MCP configuration, and a read-only tool allowlist.
- `scripts/seed_data.py` creates a deterministic 36-hour scenario with stable
  event IDs so repeated demo ingests can be deduplicated safely.
- `scripts/setup_splunk.sh` installs the app, creates `index=security`, provisions
  a dedicated search-only Splunk role and user, and mints a 30-day encrypted MCP
  token for that identity.
- The runner validates successful SDK completion and required report sections
  before atomically writing an artifact.

## Challenges we ran into

- The Splunk MCP Server accepts app-minted encrypted tokens with audience `mcp`,
  not a generic token copied from another auth flow.
- MCP tools require exact Claude SDK permission names. The correct form is
  `mcp__splunk__splunk_run_query` or an explicit wildcard such as
  `mcp__splunk__*`.
- Splunk's default local certificate cannot be trusted safely by setting a
  process-wide TLS bypass. We instead bind all ports to loopback and use local
  HTTP only for the management/MCP port in this disposable demo.
- An agent can finish with an SDK error result. Sockeye now fails closed and
  never writes an error message as if it were a completed report.

## Accomplishments we're proud of

- The agent investigates rather than following a hard-coded query sequence.
- Every run includes a machine-generated record of the exact SPL submitted to
  Splunk, independent of what the model says in prose.
- The agent token belongs to a dedicated identity scoped to `index=security`,
  rather than the Splunk administrator.
- The repository includes automated tests, linting, dependency auditing, Compose
  validation, reproducible setup, and a root-level architecture diagram.

## What we learned

MCP is valuable for agentic operations when it is treated as a security boundary,
not just a convenience layer. The useful pattern is an expiring identity, narrow
RBAC, explicit client-side tool permissions, server-side query controls, and an
execution trail a human can audit.
